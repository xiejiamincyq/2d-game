# Boss 重设计计划：深渊监工

## 1. 目标

将现有 `OVERSEER` 从“高血量通用敌人”重构为一场 45–65 秒、近战压迫与东方 Project 式弹幕躲避兼具的独立首领战。

完成后的首领必须：

- 出现时显示固定在屏幕顶部的专用血条，并清楚标记三个阶段。
- 体型和轮廓明显大于 Bruiser，拥有中距离触手武器，而不是只靠贴身碰撞。
- 远程攻击形成可读、可躲、有安全通道的弹幕图案，而不是随机散弹。
- 威胁主要来自攻击组合与阶段变化，不依赖冗长无敌或单纯堆血。
- 死亡后先清理首领弹幕和触手判定，再进入胜利结算。

本计划不改变阶段快照边界：中途退出首领战仍从第 5 阶段开场恢复，不保存首领、弹幕或攻击状态。

## 2. 已锁定的设计决策

### 2.1 登场顺序

第 5 阶段拆成两个连续段落：

1. 普通传送门攻势。
2. 普通生成队列清空且场上杂兵降至安全数量后，开启一座大型高危传送门；显示“高危目标：深渊监工”，随后生成 Boss。

Boss 战期间不生成 Marksman、Spitter 或 Lobber，避免高速弹、范围落点与首领弹幕叠成不可读攻击。阶段转换可以召唤少量 Scrapper/Dasher，数量上限为 6–8，只用于迫使玩家改变走位。

### 2.2 体型与空间

- 独立类：`OverseerBoss.gd`，不继续向通用 `Enemy.gd` 添加 Boss 分支。
- 碰撞半径：初始建议 `56px`；程序化视觉直径约 `150–170px`。
- 触手有效距离：`240–320px`，因此近战并不要求贴身。
- Boss 尽量保持在玩家 `210–340px` 范围内，并始终夹在相机可见安全区与世界边界内。
- 主体使用黑蓝/暗陶瓷材质、洋红能量核心与青色轮廓；所有危险预警统一使用警告橙 `#ff571f`。

### 2.3 生命与战斗时长

- 不先写死一个巨大生命值。由 `BalanceTest` 的“第 5 阶段标准构筑持续 DPS × 55 秒”计算首轮生命基线，预计调试区间为 `9000–12000`。
- 三段生命阈值为 `100%–70%`、`70%–35%`、`35%–0%`。
- 除登场和阶段切换各不超过 `0.65s` 的演出保护外，不使用无敌拖时。
- 标准构筑目标 TTK：`45–65s`；成熟高输出构筑允许更快，但不能跳过阶段切换演出和攻击清理。

## 3. 战斗状态机

| 状态 | 行为 | 退出条件 |
| --- | --- | --- |
| `ENTRANCE` | 大型传送门、镜头轻微聚焦、血条淡入；Boss 不攻击 | 1.1 秒演出结束 |
| `PHASE_1` | 教学型触手横扫 + 瞄准扇形弹 | 生命降至 70% |
| `TRANSITION_1` | 0.65 秒橙色脉冲，清理上一图案残弹，召唤 6 个轻型敌人 | 演出完成 |
| `PHASE_2` | 双螺旋弹幕 + 触手落点重击 | 生命降至 35% |
| `TRANSITION_2` | 0.65 秒核心过载，清理残弹，召唤 8 个轻型敌人 | 演出完成 |
| `PHASE_3` | 带安全缺口的环形弹幕 + 交替触手封路 | 生命归零 |
| `DEATH` | 停止伤害、清理 Boss 所属节点、播放碎裂演出、隐藏血条 | 清理完成后发出胜利 |

阶段切换只能触发一次。任何攻击协程或计时器在切换、暂停、死亡和退场时必须被统一取消。

## 4. 攻击设计

### 4.1 触手横扫 `Tentacle Sweep`

- 攻击前 `0.65s` 显示从 Boss 延伸出的橙色扇形/弧线，预警范围与真实判定完全一致。
- 射程 `300px`，扇形约 `78°`，伤害建议 `18`。
- 横扫过程约 `0.22s`，单次攻击对玩家最多命中一次。
- 玩家应能通过冲刺穿越、绕到 Boss 背面或退出扇形躲避。

### 4.2 触手落点 `Tentacle Slam`

- 记录玩家当前位置并生成 2–3 个直径 `90px` 的落点圈，预警 `0.75s`。
- 落点锁定后不追踪玩家；命中伤害建议 `20`。
- 每个落点爆发 6 颗低速外扩弹，迫使玩家在躲落点后继续微操。
- 落点之间必须保留至少 `70px` 通道，不能完全封死玩家。

### 4.3 瞄准扇形弹 `Aimed Fan`

- 面向玩家发射 5–7 颗扇形弹，中心方向在 `0.45s` 预警后锁定。
- 弹速 `220–280px/s`；这类弹故意慢于 Marksman，强调观察和穿缝。
- 连续两轮扇形弹之间至少保留 `0.55s` 走位窗口。

### 4.4 双螺旋弹幕 `Twin Spiral`

- 两个相对发射点以相反方向旋转，每 `0.14s` 发射一组，共 8–10 组。
- 弹速 `150–210px/s`，角速度固定，图案可学习而非随机。
- 单次图案最多生成 72 颗子弹；下一图案开始前清出明显空窗。

### 4.5 缺口环形弹幕 `Broken Ring`

- 每轮生成 20–24 颗环形弹，但在面向玩家的方向留下 `46°–60°` 安全缺口。
- 连续 3 轮时缺口逐轮旋转 `25°–35°`，形成东方 Project 式引导移动。
- 第三阶段可与触手横扫交替，但不能在安全缺口内同步生成触手伤害区。

### 4.6 公平性规则

- 同一时刻只允许一个“主弹幕图案”和一个处于预警期的触手攻击。
- Boss 弹幕全局上限建议 `140`，平均生成预算不超过 `40颗/秒`。
- 慢速弹幕、触手预警和 Marksman 高速线使用不同颜色/形状，不混用视觉语言。
- 玩家出生点、世界边缘和相机外不得成为必经安全缺口。
- 暂停、结算、死亡或 Boss 退场时，所有攻击计时停止且不能继续结算伤害。

## 5. Boss 血条与界面

- 新建 `BossHealthBar`，放在现有 `GameUI` 的 `CanvasLayer` 中，固定屏幕顶部中央。
- 基准视口 1280×720：宽约 `760px`、高 `28px`、顶部偏移 `18px`；宽度响应式限制在视口的 `60%–78%`。
- 显示名称“深渊监工 / OVERSEER”、当前阶段，以及 70%/35% 两个阈值刻度。
- 受击时只做短促亮边和损失段，不随世界镜头震动；Boss 死亡后淡出。
- 在 `960×540`、`1280×720`、`1920×1080`、`2560×1080` 下不得遮挡波次、生命、金币和暂停按钮。
- 色彩：低饱和黑蓝底、洋红生命填充、橙色阶段刻度、青色细边；避免大面积 bloom 影响弹幕阅读。

建议事件接口：

```gdscript
signal boss_spawned(boss: Node, display_name: String, maximum_health: float)
signal boss_health_changed(current: float, maximum: float, phase: int)
signal boss_defeated(boss: Node)
```

UI 只消费事件，不读取 Boss 内部攻击状态。

## 6. 代码架构

```text
WaveDirector
  ├─ final-stage regular portal attack
  ├─ Boss entrance gate
  └─ boss_spawned / boss_defeated
          │
          ▼
OverseerBoss (movement, health, phases, public signals)
  ├─ BossAttackDirector (attack scheduling and fairness budget)
  ├─ TentacleAttack (telegraph + one-hit geometry)
  └─ BossProjectilePattern (fan / spiral / broken ring)

Main
  └─ connects boss lifecycle to GameUI
          │
          ▼
BossHealthBar (screen-space only)
```

`OverseerBoss` 保持与现有击杀/伤害管线兼容的 `died`、`damage_resolved` 信号，但攻击调度和表现不塞回通用敌人脚本。Boss 子弹必须带 `boss_owner_id` 或进入专用组，以便阶段切换和死亡时只清理 Boss 弹幕。

## 7. 实施任务

### Task B1：首领契约与生成门（中型）

**内容：** 建立独立 Boss 类、生命周期信号和最终阶段两段式生成。

**验收：**

- 第 5 阶段普通攻势降至安全数量后才生成 Boss，且只生成一次。
- Boss 碰撞半径、视觉尺寸和世界/相机边界均符合设计。
- Boss 死亡前不会触发胜利，死亡清理后只触发一次。

**验证：** 新增 `BossTest` 的生成门、唯一实例、死亡顺序测试。

**依赖：** 无。

**可能文件：** `WaveDirector.gd`、`OverseerBoss.gd`、`BossTest.gd`。

**规模：** 中型，3 个文件。

### Task B2：固定 Boss 血条（中型）

**内容：** 建立响应式顶部血条并连接 Boss 生命周期。

**验收：**

- Boss 登场显示、受击更新、阶段刻度切换、死亡隐藏。
- 血条不受相机移动/震动影响，四种目标分辨率均不溢出。
- 普通阶段、继续游戏、失败和重开不会留下血条。

**验证：** 扩展 `UITest`，新增 Boss UI 生命周期测试与人工四分辨率检查。

**依赖：** B1 的信号契约。

**可能文件：** `BossHealthBar.gd/.tscn`、`GameUI.gd`、`Main.gd`、`UITest.gd`。

**规模：** 中型，4–5 个文件。

### Task B3：触手近战竖切（中型）

**内容：** 完成横扫与落点两种触手攻击，包括预警、真实判定、音画反馈和单次命中保护。

**验收：**

- 预警几何与伤害几何一致，玩家在预警结束前不受伤。
- 横扫和落点各自最多命中一次，并保留可躲通道。
- 暂停、阶段切换和死亡会取消未完成攻击。

**验证：** `BossTest` 覆盖时间窗、范围边界、重复命中和取消行为。

**依赖：** B1。

**可能文件：** `OverseerBoss.gd`、`TentacleAttack.gd`、`BossTest.gd`、战斗反馈映射。

**规模：** 中型，4 个文件。

### Checkpoint B-A：首领可读性

- Boss 能独立登场，顶部血条正确显示。
- 两种触手攻击均能通过观察预警躲开。
- 全套既有测试通过，无运行时节点泄漏。
- 人工审核 Boss 体型、轮廓、血条位置和橙色预警语言。

### Task B4：东方 Project 式弹幕控制器（中型）

**内容：** 实现瞄准扇形、双螺旋和缺口环形三种确定性图案。

**验收：**

- 相同种子生成相同图案；每个图案存在可验证安全通道。
- 弹幕上限、每秒预算和图案间空窗符合公平性规则。
- Boss 死亡和阶段切换清理全部 Boss 弹幕。

**验证：** `BossPatternTest` 检查角度分布、安全缺口、节点上限和 30/60/120Hz 一致性。

**依赖：** B1。

**可能文件：** `BossAttackDirector.gd`、`BossProjectilePattern.gd`、`BossPatternTest.gd`、`Projectile.gd`。

**规模：** 中型，4 个文件。

### Task B5：三阶段攻击编排（中型）

**内容：** 将触手与弹幕接入完整状态机，加入短转换和有限援军。

**验收：**

- 70%/35% 阶段各触发一次，攻击节奏随阶段增强。
- 不会出现“无安全缺口弹幕 + 缺口内触手伤害”的不可躲组合。
- 援军数量受限，且不含 Marksman/Spitter/Lobber。

**验证：** 状态机、攻击互斥、援军白名单和阶段跳伤测试。

**依赖：** B3、B4。

**可能文件：** `OverseerBoss.gd`、`BossAttackDirector.gd`、`WaveDirector.gd`、`BossTest.gd`。

**规模：** 中型，4 个文件。

### Task B6：音画、平衡与性能验收（中型）

**内容：** 完成登场、触手、阶段转换、弹幕和死亡反馈，并校准 TTK 与性能。

**验收：**

- 标准构筑 TTK 为 45–65 秒，伤害压力来自攻击而非长时间无敌。
- 140 颗 Boss 弹幕 + 8 个援军场景保持既有性能门，无无限 AudioStreamPlayer/VFX。
- Boss 死亡后无残留伤害、血条、相机偏移或音频循环。

**验证：** `BalanceTest`、`PerformanceTest`、`SmokeTest`、完整测试以及一次人工完整 Boss 战。

**依赖：** B2、B5。

**可能文件：** `AudioManager.gd`、`CombatFeedback.gd`、Boss 脚本、测试与文档。

**规模：** 中型，5 个文件。

### Checkpoint B-B：Boss 重设计完成

- 完整流程：第 5 阶段普通攻势 → Boss 登场 → 三阶段战斗 → 弹幕清理 → 胜利。
- 顶部血条、体型、触手近战和三种弹幕均通过人工审核。
- 四种分辨率、30/60/120Hz、暂停/继续、死亡/重开和存档边界通过。
- 完整自动测试无错误、泄漏或跳过后，再提交并推送 Boss 重构。

## 8. 依赖与并行边界

```text
B1 Boss 生命周期与信号
  ├─ B2 顶部血条（可并行）
  ├─ B3 触手攻击（可并行）
  └─ B4 弹幕图案（可并行）
          │
          └─ B5 三阶段编排（必须等待 B3+B4）
                    │
                    └─ B6 音画、平衡和性能
```

B1 信号契约锁定后，UI、触手和弹幕三个任务可以分工作树并行；`Main.gd`、`WaveDirector.gd` 和最终状态机集成必须串行合并，避免同一状态所有权冲突。

## 9. 风险与控制

| 风险 | 影响 | 控制措施 |
| --- | --- | --- |
| 弹幕与触手形成不可躲组合 | 高 | 主图案/触手调度互斥；自动验证安全缺口 |
| Boss 仍被感知为血牛 | 高 | 以标准 DPS×目标 TTK 定生命；禁用长无敌；提高攻击变化而非单纯加血 |
| 满屏节点导致卡顿 | 高 | Boss 弹幕硬上限 140、生成预算 40/s、死亡/转阶段批量清理 |
| 血条遮挡现有 HUD | 中 | 独立屏幕空间组件；四分辨率自动布局测试 |
| 通用 Enemy 脚本继续膨胀 | 中 | 独立 `OverseerBoss` 和攻击控制器，仅保留兼容信号接口 |
| Boss 中途退出恢复复杂 | 中 | 保持稳定边界语义，始终从第 5 阶段开场重建 |

## 10. 视觉资产策略

第一轮工程实现继续使用程序化 `_draw()`，先验证尺寸、轮廓、弹幕和碰撞可读性，不生成最终图片。若后续替换为正式 Boss 插画，必须按 `godot-neon-art-pipeline` 单独建立 manifest、锁定角色/战斗批次风格，并由用户审核透明边缘、顶视角、中心轴、触手外溢和实际 150–170px 尺寸表现；美术替换不得改变碰撞和伤害半径。

# 战斗手感优化技术实施计划

## 1. 目标与范围

把现有战斗循环收敛为：

```text
高速扫射 -> 冲刺切入 -> 击杀敌群 -> 返还冲刺冷却 -> 继续扫射
```

本轮同时改善射击空、移动飘和冲刺不爽，但严格按依赖顺序实施：先建立反馈与移动地基，再完善冲刺闭环，然后调整敌人和升级。每个任务完成后游戏都应保持可运行、可测试，不能等到最后才把系统接起来。

计划保留项目现有的程序化绘制与合成音频路线，不引入外部素材包、新物理框架或通用能力状态机。

## 2. 当前代码约束

- `Player.gd` 直接把输入向量乘以 `move_speed`，逻辑上没有明显惯性；“飘”更可能来自相机平滑速度 `8.0`，以及缺少转向、起步、刹停的视觉响应。
- 主武器基础射速为每秒 13 发，升级后还会显著提高。不能为每一发普通命中触发全局 hit-stop，也不能为每次命中创建多个临时节点。
- 敌人命中反馈目前主要是 `0.08s` 白闪和限频音效，死亡立即 `queue_free()`。
- 冲刺当前使用分段扫掠判定敌人，并直接推进位置；没有无敌、输入缓存、残影、冷却提示或击杀返还。
- `WorldBoundary` 目前只有绘制，没有实体障碍。因此冲刺继续使用扫掠判定与矩形边界钳制，不为尚不存在的地形碰撞过度设计。
- `DASHER` 主要是更快、更脆，非远程敌人基本共用同一套近战前摇和恢复逻辑。
- 升级系统以数值倍率为主，缺少冲刺与武器/被动能力之间的机制联动。

## 3. 可验证的体验指标

- 移动从静止到满速不超过 `50ms`，松开输入到停止不超过 `40ms`，反向切换不超过 `70ms`。
- 相机对比关闭平滑、平滑速度 `16` 和 `20`，选择快速反向移动时最不拖拽的方案。
- 普通子弹命中不触发全局 hit-stop；子弹击杀最多 `12ms`，冲刺命中最多 `18ms`，冲刺击杀或重装打断最多 `25-30ms`。
- 任意滚动 `100ms` 窗口内，合并后的 hit-stop 总量不超过 `35ms`。
- 冲刺初始保持 `165px / 0.16s`，无敌只覆盖有效位移窗口，结束后第一帧即可恢复或消费缓存射击。
- 基础冲刺击杀返还先以每个敌人 `0.45s`、单次冲刺最多 `1.2s`、至少保留 `0.4s` 冷却为测试起点。
- 150 个以上活动敌人时，战斗特效数量仍有明确上限，不出现“每次命中永久增加节点”的路径。

## 4. 架构设计

### 4.1 战斗反馈协调器

新增 `scripts/systems/CombatFeedback.gd`，由 `Main` 创建和连接。角色与武器只汇报战斗事实，不直接操作相机、UI 或全局时间。

建议公开接口：

```gdscript
func request_hit_feedback(
    source: StringName,
    world_position: Vector2,
    direction: Vector2,
    killed: bool,
    intensity: float = 1.0
) -> void

func request_player_damage(
    world_position: Vector2,
    direction: Vector2,
    shield_absorbed: bool
) -> void

func request_dash_afterimage(
    world_position: Vector2,
    facing: float
) -> void

func clear_all_feedback() -> void
```

这不是全局事件总线。`Main` 继续承担组装职责，符合当前项目结构，也便于测试时替换或直接调用。

### 4.2 伤害事件契约

保留现有 `Enemy.hit(source)` 信号供音效使用，新增更完整的事件：

```gdscript
signal damage_resolved(
    enemy: Node,
    source: StringName,
    amount: float,
    world_position: Vector2,
    direction: Vector2,
    killed: bool
)
```

兼容扩展敌人伤害入口：

```gdscript
func take_damage(
    amount: float,
    source: StringName = DamageTypes.GENERIC,
    hit_direction: Vector2 = Vector2.ZERO
) -> bool
```

返回值表示这次被接受的伤害是否造成死亡。旧调用者可以继续只传两个参数并忽略返回值；方向性来源在各自任务中补充方向，范围伤害允许使用 `Vector2.ZERO`。

玩家新增信号：

```gdscript
signal damage_received(amount: float, shield_absorbed: float, source: StringName)
signal dash_started(direction: Vector2)
signal dash_ended
signal dash_cooldown_changed(remaining: float, maximum: float)
signal dash_kill(enemy: Node)
```

### 4.3 程序化特效与性能边界

新增一个 `scripts/effects/CombatVfx.gd`，用有容量上限的数组记录火花、碎片、冲击环和残影，在单个 `_draw()` 中绘制：

- 枪口焰保留在 `Player._draw()`，只增加短计时器。
- 每颗子弹最多保存 4 个历史位置，在自身 `_draw()` 中绘制短拖尾。
- 命中火花、死亡碎片、冲刺残影进入统一 VFX 数组，不创建一批长期节点。
- 每类效果都有容量；达到上限时优先丢弃最旧、优先级最低的记录。
- 暂停、升级、结算时不继续生成可见效果，重开与退树时清空全部记录。

### 4.4 相机冲击

新增 `scripts/effects/CameraEffects.gd` 挂到运行时创建的 `Camera2D`：

```gdscript
func add_impulse(strength: float, direction: Vector2 = Vector2.ZERO) -> void
func reset_immediately() -> void
```

冲击应合并并钳制最大位置/旋转偏移，衰减结束后精确回到零。普通子弹命中只允许极轻或无相机冲击，玩家受伤、冲刺击杀、重装打断使用更高等级。

### 4.5 Hit-stop 生命周期

`CombatFeedback` 使用 `PROCESS_MODE_ALWAYS`。如果降低 `Engine.time_scale`，恢复截止时间必须基于 `Time.get_ticks_msec()`，不能使用缩放后的 `delta`。时间倍率不能设为 0，并必须在以下路径恢复为 `1.0`：

- hit-stop 到期；
- 进入升级、暂停或结算；
- 重开场景；
- `CombatFeedback._exit_tree()`。

只有击杀、冲刺重击、重装打断和玩家受伤可以请求全局 hit-stop。

### 4.6 玩家移动与冲刺

不引入通用状态机框架。`Player.gd` 保持明确的普通移动与冲刺两种路径，并提取可测试的小函数：

```gdscript
func _update_normal_movement(delta: float, input_vector: Vector2) -> void
func _consume_dash_input() -> bool
func refund_dash_cooldown(seconds: float) -> float
func get_dash_cooldown_ratio() -> float
```

冲刺使用现有的线段扫掠保证低帧率下不漏敌人，并继续通过世界边界钳制位置。输入缓存只负责避免动作衔接吞键，不自动连发多个动作。

### 4.7 核心能力与升级的边界

以下属于基础可用性，不应被升级锁住：

- 冲刺有效窗口无敌；
- 冲刺结束立即恢复射击；
- 冲刺冷却可读；
- 保守的基础击杀返还。

升级只扩大或组合这些能力，例如提高返还、冲刺后短暂超频、被动击杀返还；不能用升级修复基础操作缺陷。

## 5. 依赖图

```text
伤害/反馈事件契约
  -> CombatFeedback + CameraEffects + CombatVfx
    -> 射击命中/死亡反馈
    -> 玩家受伤反馈
    -> 冲刺音画反馈

移动响应测量 + 相机调校
  -> 冲刺状态与输入连续性
    -> 冲刺击杀返还
      -> 敌人打断交互
      -> 冲刺/武器联动升级

敌人阶段辅助逻辑
  -> 疾冲者前摇/冲刺/恢复
  -> 吐弹者预警/发射
  -> 重装者前摇/打断/恢复

所有纵向切片
  -> 八波平衡、性能与最终回归
```

## 6. 分阶段任务

### Task 1：建立伤害事件与反馈协调器骨架

**目的：** 建立兼容的伤害结果信号、反馈请求入口和 hit-stop 生命周期，但暂不加入复杂视觉。

**验收标准：**

- [ ] 每次被接受的敌人伤害只产生一次结果事件，包含正确来源、实际伤害、死亡状态和可选方向。
- [ ] 零伤害或未接受伤害不产生反馈；旧 `hit(source)` 音效路径保持一次命中一次信号。
- [ ] hit-stop 请求合并、优先级和滚动上限可重复验证；所有退出路径都恢复 `Engine.time_scale == 1.0`。

**验证：**

- 新增 `CombatFeedbackTest.gd`，覆盖事件载荷、死亡返回值、请求合并、上限和清理。
- 单独运行 `CombatFeedbackTest`、`DamageTest`、`StateTest`。
- 运行完整严格测试脚本。

**依赖：** 无

**预计文件：** `CombatFeedback.gd`、`Enemy.gd`、`Main.gd`、`CombatFeedbackTest.gd`、`run_tests.ps1`

**规模：** M

### Task 2：实现有边界的程序化 VFX 与相机冲击

**目的：** 建立统一 VFX 绘制器和相机效果组件，并由反馈协调器路由。

**验收标准：**

- [ ] 每类 VFX 有明确容量和生命周期，过期后数组归零，不遗留节点。
- [ ] 多个相机冲击可预测地合并，始终受最大偏移/旋转限制，结束后回到精确零值。
- [ ] 暂停/升级/结算时不新增可见反馈，但清理逻辑仍能运行。

**验证：**

- 扩展 `CombatFeedbackTest` 检查容量淘汰、生命周期和相机归零。
- 人工触发弱/中/强三档请求，检查 1280×720 下的可读性。
- 运行 `StateTest`、`UITest`、`PerformanceTest` 和完整测试。

**依赖：** Task 1

**预计文件：** `CombatVfx.gd`、`CameraEffects.gd`、`CombatFeedback.gd`、`Main.gd`、`CombatFeedbackTest.gd`

**规模：** M

### Task 3：量化并收紧移动与相机响应

**目的：** 分离普通移动更新，使用快速加速/减速参数，同时对比相机平滑设置；只增加视觉倾斜/收势，不改变碰撞体积。

**验收标准：**

- [ ] 30/60/120 Hz 下，起步、停止、反向时间均达到体验指标，误差不超过一个模拟帧。
- [ ] 对角移动保持归一化，世界边界行为不变。
- [ ] 快速左右反向时没有明显相机拖拽，最终选择值和理由写在常量附近。

**验证：**

- 新增 `MovementTest.gd`，验证三种帧率下的响应时间和对角速度。
- 人工 A/B 对比关闭平滑、`16`、`20`。
- 运行 `MovementTest`、`DashTest`、`RateTest` 和完整测试。

**依赖：** 无；可以与 Task 1 独立实施，但在事件契约确定后再合入。

**预计文件：** `Player.gd`、`Main.gd`、`MovementTest.gd`、`run_tests.ps1`

**规模：** M

### Checkpoint A：响应地基

- [ ] 完整严格测试通过，无错误或泄漏标记。
- [ ] 相机、VFX、时间倍率在暂停、升级、结算、重开后均回到中性状态。
- [ ] 连续 10 分钟移动测试无明显拖拽或漂移感。
- [ ] 在加入更多反馈前人工确认相机和移动参数。

### Task 4：完成主武器射击反馈闭环

**目的：** 增加第一帧枪口焰/后坐表现、短弹道拖尾、方向性火花和克制的击杀反馈，不改变射速算法。

**验收标准：**

- [ ] 按下射击后一个模拟帧内出现枪口反馈，投射物生成节奏与数量保持原有帧率独立性。
- [ ] 命中火花方向正确；子弹击杀才允许受限 hit-stop、碎片和较明显相机冲击。
- [ ] 五枪线和高射速叠加时特效数量仍有上限，音效仍受现有限频控制。

**验证：**

- 扩展 `RateTest`，证明视觉计时器不影响十秒射击数量。
- 扩展 `ProjectilePickupTest`，验证命中方向与拖尾容量。
- 人工检查基础武器、五枪线、高射速三种状态。
- 运行 `RateTest`、`ProjectilePickupTest`、`PerformanceTest` 和完整测试。

**依赖：** Tasks 1-3

**预计文件：** `Player.gd`、`Projectile.gd`、`CombatFeedback.gd`、`RateTest.gd`、`ProjectilePickupTest.gd`

**规模：** M

### Task 5：增加敌人受击与死亡反馈

**目的：** 让敌人受击方向和死亡碎片都有清晰但受控的反馈。玩法死亡和奖励结算仍应立即发生，表现层不能延迟波次逻辑。

**验收标准：**

- [ ] 敌人死亡时经验、护盾掉落和波次剩余只结算一次；表现可以延续 `80-150ms`，但碰撞与玩法对象立即失效。
- [ ] 受击方向与来源一致，敌人视觉位移不会越过世界边界。
- [ ] 子弹、激光、电弧、地刺和冲刺都遵守统一的敌人反馈强度映射。

**验证：**

- 扩展 `WaveTest` 检查单次死亡、奖励和活动敌人移除。
- 人工检查子弹、激光、电弧、地刺、冲刺五种来源。
- 运行 `WaveTest`、`CombatFeedbackTest`、`BalanceTest`、`PerformanceTest` 和完整测试。

**依赖：** Tasks 1、2、4

**预计文件：** `Enemy.gd`、`WaveDirector.gd`、`CombatVfx.gd`、`WaveTest.gd`、`CombatFeedbackTest.gd`

**规模：** M

### Task 6：完成玩家受伤与护盾反馈

**目的：** 区分伤害被拒绝、护盾吸收、护盾破裂和生命受伤，增加克制的红/青闪光、相机冲击和音效，同时保持既有伤害与无敌帧语义。

**验收标准：**

- [ ] 护盾吸收和生命损失产生不同的已接受伤害载荷与反馈；无敌帧拒绝的伤害保持静默。
- [ ] 护盾破裂只触发一次更强反馈，但不改变护盾溢出到生命的伤害算术。
- [ ] 既有 `0.35s` 受击后无敌时间和接受序列保持不变，拒绝伤害不会刷新计时器。

**验证：**

- 扩展 `DamageTest` 检查信号次数、护盾吸收量、护盾破裂边界和拒绝伤害。
- 人工检查纯护盾伤害、破盾溢出、纯生命伤害和高频接触伤害。
- 运行 `DamageTest`、`StateTest` 和完整测试。

**依赖：** Tasks 1、2

**预计文件：** `Player.gd`、`CombatFeedback.gd`、`AudioManager.gd`、`DamageTest.gd`

**规模：** S-M

### Checkpoint B：射击与伤害手感

- [ ] 密集火力仍流畅，普通命中不会造成常驻减速。
- [ ] 五类伤害来源可区分，但共用一致的反馈强度规则。
- [ ] `PerformanceTest` 不低于既有容差。

## 8. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 高频射击导致常驻 hit-stop | 高 | 普通命中不暂停；请求合并；滚动上限；高射速回归 |
| `Engine.time_scale` 未恢复 | 高 | 真实时间截止、始终处理、状态转换与退树显式清理 |
| 死亡表现延迟奖励或波次结束 | 高 | 玩法死亡与表现分离；单次奖励/移除测试 |
| 冲刺返还形成永久无敌 | 高 | 单杀/单次上限、最低冷却、无基础完全刷新 |
| 震屏遮挡敌人前摇 | 中 | 方向性低振幅、最大偏移、表现强度配置 |
| 程序化特效造成分配尖峰 | 中 | 单一有界 VFX 渲染器、固定拖尾长度、低优先级淘汰 |
| 临时射速导致永久属性漂移 | 高 | 基础值与有效值分离，刷新/过期确定性测试 |
| 敌人阶段逻辑帧率相关 | 中 | 30/60/120 Hz 阶段和位移测试 |
| 平衡调整掩盖手感回归 | 中 | 最后单独调参；一次只改一类参数 |

## 9. 明确不做

- 不建立通用战斗能力/状态机框架。
- 不为当前不存在的物理障碍设计复杂冲刺碰撞。
- 不为每发子弹添加全局 hit-stop。
- 不增加独立护甲生命系统，先用冲刺打断验证重装者身份。
- 不加入格斗游戏式连招树和复杂取消窗口。
- 不一次增加大量新武器或敌人。
- 不引入外部美术/音频包。

## 10. 检查点待决参数

- 相机平滑关闭、`16`、`20` 中哪一个最跟手。
- 基础冲刺返还是否足以改变玩家行为，又不会让相关升级变成必选。
- 重装者仅靠打断/恢复能否形成身份，还是后续确实需要伤害抗性。
- 是否需要默认暴露震屏/闪光选项，或仅在测试显示不适时加入设置界面。
